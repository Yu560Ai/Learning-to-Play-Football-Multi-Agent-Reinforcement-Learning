## What `extended_plot_summary.json` Summarizes

For 5M `r3_assist`, `extended_plot_summary.json` does **not** summarize deterministic evaluation.

It is generated by `plot_phase1_extended.py`, which:

- reads `metrics.jsonl`
- takes the raw last finite value for each metric
- writes that value into `extended_plot_summary.json`

So the 5M `r3_assist` entry comes from the last line of:

- `Two_V_Two/results/phase1_extended/budget_5000000/r3_assist/metrics.jsonl`

That last line is:

- `update = 3125`
- `env_steps = 5000000`
- `mean_goal_count = 0.01`
- `mean_pass_count = 0.03`
- `mean_pass_to_shot_count = 0.01`

These are training-time metrics computed from the recent completed training episodes during the on-policy run. They are not deterministic eval numbers.

Important detail:

- the plotted curves use smoothing for visualization
- but `extended_plot_summary.json` itself stores the last raw metric value, not a smoothed value

So this is **not** a smoothing-window mismatch.

## What the Final Deterministic Eval Summarizes

The 5M deterministic eval is produced by:

- `Two_V_Two/evaluation/run_phase1_eval.py`

For 5M `r3_assist`, it loads:

- `Two_V_Two/results/phase1_extended/budget_5000000/r3_assist/checkpoints/latest.pt`

and runs:

- `50` deterministic evaluation episodes

The resulting entry in:

- `Two_V_Two/results/phase1_extended/budget_5000000/eval_summary.json`

reports:

- `checkpoint = latest.pt`
- `checkpoint_update = 3125`
- `checkpoint_env_steps = 5000000`
- `goal_rate = 0.0`
- `mean_goal_count = 0.0`
- `mean_pass_count = 0.0`
- `mean_pass_to_shot_count = 0.0`
- `mean_assist_count = 0.0`

## Are These the Same Checkpoint?

Not exactly in the way the question was framed:

- `extended_plot_summary.json` is **not** a checkpoint summary
- it is a summary of the last training-log row from `metrics.jsonl`

The deterministic eval does summarize a checkpoint:

- `latest.pt`

However, both refer to the same final training step:

- `update = 3125`
- `env_steps = 5000000`

I also verified that:

- `latest.pt` and `update_003125.pt` have the same payload content for the actor weights
- both report `update = 3125` and `env_steps = 5000000`

So the discrepancy is **not** due to evaluating a different final model checkpoint.

## Why the Numbers Differ

The numbers differ because they come from different measurement pipelines:

1. `extended_plot_summary.json`
- source: training log tail from `metrics.jsonl`
- action mode: stochastic training-time action sampling
- aggregation: recent completed training episodes

2. `eval_summary.json`
- source: separate deterministic evaluation run
- action mode: deterministic/greedy actions
- aggregation: fresh evaluation episodes

So the discrepancy is real in the sense that the two files report different values, but it is a **bookkeeping / metric-source mismatch**, not a contradictory checkpoint identity.

## Corrected Interpretation

- The final 5M `r3_assist` policy did show nonzero pass-level behavior in the training log at the end of training.
- But when the same final learned policy was evaluated deterministically, that pass-level behavior did not survive.
- Therefore the correct reading is:
  - `R3` is behaviorally different from `R2` during training
  - that difference is unstable
  - the final 5M deterministic checkpoint does **not** provide reproducible pass-based separation from `R2`

## Sweep Decision

No additional checkpoint sweep was run in this verification step.

Reason:

- the discrepancy was resolved directly from the artifact chain
- it is explained by training-log metrics vs deterministic eval metrics
- it is not caused by mixing different runs or different final checkpoints
